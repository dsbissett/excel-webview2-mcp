# Plan: Connection Robustness for Excel WebView2 MCP

**Target:** Make the MCP server resilient to the realities of connecting to a user-launched Excel WebView2 session — add health probes, retry/backoff on initial connect, mid-session reconnect, a runtime status tool, better error messages, and launch-instruction docs. No change to tool surface semantics; this is purely a quality-of-life layer around connection lifecycle.

**Author context:** Created 2026-04-18. Each phase is self-contained and executable in a fresh chat.

### What's missing (target state)

1. Pre-connect probe with configurable timeout.
2. Retry-with-backoff on initial connect (opt-in, finite).
3. `browser.on('disconnected')` handler that either reconnects or marks context as stale with a clear error.
4. A `connection_status` tool so Claude can ask "am I connected?" without guessing.
5. Error messages that name Excel WebView2, cite port 9222, and suggest the exact verification command.
6. Launch-instruction docs for enabling the WebView2 debug port in an Office add-in.

### Anti-patterns to guard against

- **Do NOT retry indefinitely.** Always bound retries with a total budget (count or time).
- **Do NOT swallow the original error.** Wrap with `cause` so debug users see the Puppeteer-level failure.
- **Do NOT hide the hardcoded 9222 default behind new abstraction layers** — keep the one source of truth at `WEBVIEW2_DEBUG_URL` and just reference it.
- **Do NOT add a background keepalive ping that spams the CDP endpoint.** A disconnect event handler is cheaper and more accurate than polling.
- **Do NOT auto-launch Excel.** The project explicitly does not manage the add-in; don't drift into launching it.
- **Do NOT edit `docs/tool-reference.md` by hand** — it's regenerated by `npm run gen`.

## Phase 1 — Pre-Connect Health Probe

**Goal:** Before calling `puppeteer.connect`, probe `/json/version` at the target URL and fail fast with a precise, Excel-aware message if it's unreachable.

### What to implement

1. Add a small helper `probeCdpEndpoint(url: string, timeoutMs: number): Promise<{ok: boolean, version?: string, reason?: string}>` in a new file `src/connection/probe.ts`.
   - GET `${url}/json/version` using Node's built-in `fetch` with `AbortSignal.timeout(timeoutMs)`.
   - Return `{ok: true, version: body.Browser}` on 2xx with parseable JSON.
   - Return `{ok: false, reason: 'unreachable' | 'timeout' | 'invalid-response' | 'http-error:<status>'}` otherwise.
   - Never throw — always return a result.

2. In [src/browser.ts](../../src/browser.ts), call `probeCdpEndpoint` immediately before `puppeteer.connect` when the resolved endpoint is an HTTP URL (browserURL mode). Skip the probe for `wsEndpoint` mode — the CDP version endpoint is HTTP-only.

3. If the probe returns `{ok: false}`, throw a typed `ConnectionError` with:
   - The probe reason.
   - The resolved URL.
   - The one-line hint: `Run: curl ${url}/json/version — if this fails, your Excel add-in isn't exposing the debug port.`
   - Original cause preserved (wrap with `{cause}`).

4. Add a new CLI flag `--connectTimeout` (milliseconds, default `5000`) in [src/bin/excel-webview2-mcp-cli-options.ts](../../src/bin/excel-webview2-mcp-cli-options.ts). Thread it through to `probeCdpEndpoint`.

### Documentation references

- CDP `/json/version` shape: `docs/plans/connection-robustness-references.md` (from Phase 0).
- CLI option pattern: existing `--browserUrl` / `--wsEndpoint` definitions in the same file.
- Error wrapping convention: [src/browser.ts:141-143](../../src/browser.ts#L141-L143) (`new Error(msg, {cause: err})`).

### Verification checklist

- Unit test for `probeCdpEndpoint`: mock fetch for the four return cases (ok, timeout, unreachable, http-error). Place in `tests/connection/probe.test.ts`.
- `npm run build` succeeds.
- `npm run format` clean.
- Manual: with Excel add-in NOT running, start the server and invoke any tool — confirm the error message now includes the URL and the `curl` hint, not the generic "Could not connect to Chrome" text.
- Manual: with add-in running, confirm the probe adds only a few ms of startup latency.

### Anti-pattern guards

- Do not add the probe inside a retry loop yet (Phase 2 owns retry).
- Do not `console.log` the probe result on success — that's noise.
- Do not require the user to opt in via a flag — the probe is always-on in browserURL mode.

## Phase 2 — Retry with Backoff on Initial Connect

**Goal:** Make initial connect tolerate a slow-starting Excel add-in without requiring the user to time a race.

### What to implement

1. Add a retry wrapper `connectWithRetry(options)` in `src/connection/retry.ts`:
   - Runs `probeCdpEndpoint` + `puppeteer.connect` together.
   - On failure, wait with exponential backoff (500ms, 1s, 2s, 4s — cap at 5s) until total elapsed time exceeds `--connectRetryBudget` (default `15000` ms).
   - Return the connected browser or throw the final `ConnectionError`.
   - Emit a `debug('mcp:log')` line per retry attempt with the attempt number and reason.

2. Add two new CLI flags:
   - `--connectRetryBudget <ms>` (default `15000`, set to `0` to disable retries).
   - `--connectRetryVerbose` (default `false`) — when true, also log retry attempts to stderr so users see progress.

3. Call `connectWithRetry` from `ensureBrowserConnected` instead of `puppeteer.connect` directly.

### Documentation references

- Probe helper signature from Phase 1.
- CLI option patterns in [src/bin/excel-webview2-mcp-cli-options.ts](../../src/bin/excel-webview2-mcp-cli-options.ts).

### Verification checklist

- Unit test for `connectWithRetry`: assert total attempts respect the budget; assert backoff delays are bounded; assert success short-circuits.
- Manual: with Excel not running, start a tool call; within the budget, start Excel; confirm the server connects without a restart.
- Manual: with `--connectRetryBudget 0`, confirm the old fail-fast behavior is preserved.

### Anti-pattern guards

- Do not retry forever.
- Do not retry on clearly terminal errors (e.g. `http-error:403`) — only on `unreachable` / `timeout`.
- Do not introduce jitter unless Phase 0 research shows a specific need — keep the backoff deterministic and testable.

## Phase 3 — Mid-Session Disconnect Handling

**Goal:** When the CDP socket drops (user closes Excel, add-in reloads), detect it and either reconnect transparently or mark the context stale so the next tool call returns an actionable error instead of a Puppeteer stack trace.

### What to implement

1. In [src/browser.ts](../../src/browser.ts) after a successful `puppeteer.connect`, wire `browser.on('disconnected', () => { ... })` to:
   - Set a module-level `connectionStale: true` flag.
   - Log via `debug('mcp:log')`: `CDP connection lost; next tool call will attempt reconnect.`

2. In `ensureBrowserConnected`, check `connectionStale` at entry: if true, discard the cached browser and run the full probe + connect + retry path again (reusing Phase 2).

3. Add a **session-level reconnect cap**: a counter for "reconnects within the last 60s"; if >3, refuse further reconnects for the rest of the session and return a sticky error `Connection is unstable — restart the MCP server.` This prevents a boot-loop masking a broken setup.

4. Ensure the error surface at [src/index.ts:249-263](../../src/index.ts#L249-L263) receives a clearly-typed `ConnectionError` so the message reaching the MCP client is clean.

### Documentation references

- Puppeteer `disconnected` event semantics: Phase 0 reference doc.
- Error propagation: [src/index.ts:249-263](../../src/index.ts#L249-L263).

### Verification checklist

- Unit test: simulate a `disconnected` emit on a mock browser; assert the flag flips and the next connect attempt runs the probe again.
- Manual: while a session is live, quit Excel entirely; run any tool; confirm the error message matches the new `ConnectionError` shape, not a raw Puppeteer error.
- Manual: restart Excel within the budget; run the tool again; confirm reconnection succeeds.
- Manual: force rapid restarts to trigger the reconnect cap; confirm the "restart the MCP server" sticky error fires.

### Anti-pattern guards

- Do not reconnect eagerly on the `disconnected` event — wait for the next tool call. Reconnecting eagerly races with the user's launch timing and generates log noise.
- Do not leave dangling Puppeteer listeners when the session ends. Remove the listener in any shutdown path.

## Phase 4 — `connection_status` Tool

**Goal:** Give Claude (and the user via CLI) an explicit way to ask: "Is the server attached? What is it attached to?" Without this, a failed tool call is the only feedback mechanism.

### What to implement

1. Create `src/tools/connection.ts` (new file) exporting one tool:
   - Name: `connection_status`, `category: ToolCategory.DEBUGGING` (or `EXCEL` if Phase 2 of the Excel-tooling plan has already landed).
   - Schema: `{ probe?: boolean }` — when true, re-run `probeCdpEndpoint` instead of returning the cached state.
   - Handler returns a JSON object:
     ```
     {
       attached: boolean,
       endpointUrl: string,          // resolved URL actually in use
       endpointSource: 'browserUrl' | 'wsEndpoint' | 'autoDetect' | 'default',
       cdpVersion: string | null,    // from /json/version
       lastProbeResult: 'ok' | 'unreachable' | 'timeout' | 'invalid-response' | 'http-error:<n>' | null,
       reconnectCount: number,       // since process start
       staleSince: string | null     // ISO timestamp if stale, else null
     }
     ```
2. Register the tool in [src/tools/tools.ts](../../src/tools/tools.ts).

### Documentation references

- Tool pattern: [src/tools/snapshot.ts:12-41](../../src/tools/snapshot.ts#L12-L41) (handler shape).
- Registration: existing imports and `...Object.values(...)` spread in `createTools()`.

### Verification checklist

- `npm run gen` regenerates `docs/tool-reference.md` with `connection_status`.
- Manual: invoke with Excel running → `attached: true`, `lastProbeResult: 'ok'`.
- Manual: invoke with Excel stopped → `attached: false`, `lastProbeResult: 'unreachable'`.
- Manual: invoke with `probe: true` after a disconnect — confirm it runs a fresh probe instead of returning stale cached state.

### Anti-pattern guards

- Do not include Puppeteer-internal objects in the response.
- Do not expose internal timers or retry state beyond the named fields — the shape is a stable contract.

## Phase Sequencing & Branching Advice

- Phase 0 is strictly first — implementation phases cite its reference file.
- Phases 1 → 2 → 3 are strictly sequential (each builds on the previous connection code path).
- Phase 4 depends on Phases 1–3 so `connection_status` can report their state.
- Phase 5 can land after Phase 1 (once `ConnectionError` makes sense) and is required before Phase 6 can cite a stable docs anchor.
- Phase 6 depends on Phase 0 for verified launch instructions.
- Phase 7 depends on everything.

If time-boxed, Phases 0 + 1 + 5 + 6 alone deliver 80% of the user-visible value (probe, typed errors, launch docs) without the reconnect machinery. Phases 2 + 3 + 4 can follow in a second pass.

## Known Gaps / Confidence Notes

- **Microsoft launch mechanism** for WebView2 debug port in Office add-ins is not confirmed here — Phase 0 must verify whether `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS=--remote-debugging-port=9222` is the supported path or whether a registry key is required. If both fail, the plan still ships but Phase 6 documents the empirically-known workaround with a caveat.
- **Reconnection budget** (3 in 60s) is a guess. Tune after real usage data.
- **CDP WebSocket disconnect behavior** on a paused-Excel (not quit) is unverified. Phase 0 should confirm whether Puppeteer emits `disconnected` on pause or only on quit.
