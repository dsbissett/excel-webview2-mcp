# Plan: Launch Excel Add-in with WebView2 Remote Debugging from MCP

**Target:** Give this MCP server the ability to spawn Excel with a sideloaded add-in and remote debugging enabled on port 9222 ŌĆö directly from the agent ŌĆö when the user is working inside an Excel add-in repository. Modeled on the user's `npm run start:cdp` script:

```jsonc
"start:cdp": "cross-env WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS=\"--remote-debugging-port=9222\" office-addin-debugging start manifest.xml"
```

The MCP server currently _connects_ to an already-running CDP endpoint (default `localhost:9222`). It does not launch Excel. This plan adds a launch capability that wraps `office-addin-debugging start` with the correct `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS` environment variable and a configurable port.

**Author context:** Created 2026-04-19. Each phase is self-contained and can be executed in a fresh chat. Cite the Discovered Facts section rather than re-deriving.

## Discovered Facts (copy into every phase's context)

### Connection vs launch boundary

- The server today only _connects_. `ensureBrowserConnected` in [src/browser.ts](../../src/browser.ts) supports `browserURL`, `wsEndpoint`, `channel`, and `userDataDir`, with default `http://localhost:9222` for WebView2.
- There is no existing process-spawn code path for Excel. This plan introduces one ŌĆö it must NOT replace the connect path; launch and connect are separate concerns wired in sequence.

### How the user launches today

- `office-addin-debugging start manifest.xml` (from the `office-addin-debugging` npm package) sideloads a manifest, starts the dev server, and opens Excel with the add-in.
- Remote debugging is enabled by setting `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS=--remote-debugging-port=9222` in the Excel process environment before launch. WebView2 reads this on startup.
- `cross-env` is only needed for cross-platform shell env-var syntax in npm scripts; when spawning from Node we set `env` directly and don't need `cross-env`.

### Detecting an Excel add-in repo

Heuristics, in order of confidence:

1. A `manifest.xml` (or `manifest.json` for Unified manifest) at repo root whose contents include `OfficeApp` / `<Host Name="Workbook">` or the unified-manifest `extensions[].requirements.scopes` includes `mail`/`workbook`.
2. `package.json` `devDependencies` includes `office-addin-debugging` or `office-addin-manifest`.
3. A `start:cdp` (or similar) script already present in `package.json`.

### Tool registration pattern

- Tools live in `src/tools/*.ts`, defined via `defineTool(...)` from [src/tools/ToolDefinition.ts](../../src/tools/ToolDefinition.ts). New tool modules are imported and spread in [src/tools/tools.ts](../../src/tools/tools.ts) inside `createTools()`.
- No repo-wide license header requirement applies to new `.ts` source files.
- Categories are defined in [src/tools/categories.ts](../../src/tools/categories.ts). A new `LIFECYCLE` category fits this work; if added, update both the enum and any docs generators that switch on category.
- Docs (`docs/tool-reference.md`, `docs/slim-tool-reference.md`) are auto-generated by `npm run gen` ŌĆö never hand-edit.

### CLI options pattern

- CLI options for the server live in [src/bin/excel-webview2-mcp-cli-options.ts](../../src/bin/excel-webview2-mcp-cli-options.ts) and `ParsedArguments` flows into `createTools(args)`.
- Connection retry/backoff CLI options were added recently (memory IDs 232, 233) ŌĆö follow that same pattern for `--launch-*` flags.

### Test patterns

- Unit tests: `tests/*.test.ts`, `node:test`. Mock `child_process.spawn` rather than actually launching Excel.
- E2E tests in `tests/e2e/` use `runCli` and require a real CDP endpoint ŌĆö do NOT add an E2E test that launches Excel in CI.

## Phase 1 ŌĆö Repo detection helper

**Goal:** A pure function that, given a directory, returns whether it looks like an Excel add-in project and what launch command should be used.

**Files to create:**

- `src/launch/detectAddin.ts` ŌĆö exports `detectExcelAddin(cwd: string): Promise<AddinProject | null>` returning:
  ```ts
  type AddinProject = {
    root: string; // absolute path
    manifestPath: string; // absolute path to manifest.xml or manifest.json
    manifestKind: 'xml' | 'json'; // unified vs classic
    packageManager: 'npm' | 'pnpm' | 'yarn'; // inferred from lockfile
    existingCdpScript?: string; // name of a script that already sets remote-debugging-port, if any
  };
  ```
- `tests/launch/detectAddin.test.ts` ŌĆö fixtures for: classic XML manifest, unified JSON manifest, non-add-in repo, repo with `start:cdp` already present.

**Detection logic:**

1. Walk up from `cwd` to find `package.json` (cap at 5 levels).
2. Read `manifest.xml` or `manifest.json` next to it; if XML, regex-check for `<OfficeApp` and `Workbook` host. If JSON, parse and look for the `extensions` array with workbook scope.
3. Inspect `scripts` in `package.json` for any value containing `--remote-debugging-port` or `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS`.
4. Detect package manager via lockfile: `pnpm-lock.yaml` ŌåÆ pnpm, `yarn.lock` ŌåÆ yarn, else npm.

**No tool registration in this phase** ŌĆö this is foundation only. Validate with `npm run test`.

## Phase 2 ŌĆö Launch executor

**Goal:** Spawn Excel with the add-in loaded and remote debugging enabled. Pure executor ŌĆö no MCP wiring yet.

**Files to create:**

- `src/launch/launchExcel.ts` ŌĆö exports:
  ```ts
  type LaunchOptions = {
    project: AddinProject;
    port: number; // default 9222
    extraBrowserArgs?: string[]; // appended to --remote-debugging-port=...
    timeoutMs?: number; // default 60_000 ŌĆö wait for CDP endpoint to come up
    signal?: AbortSignal;
  };
  type LaunchResult = {
    pid: number;
    cdpUrl: string; // http://localhost:<port>
    stop: () => Promise<void>; // runs `office-addin-debugging stop <manifest>`
  };
  async function launchExcel(opts: LaunchOptions): Promise<LaunchResult>;
  ```

**Behavior:**

1. Resolve the launcher binary. Prefer the project-local one: `<root>/node_modules/.bin/office-addin-debugging`. Fall back to `npx --no-install office-addin-debugging` if absent. Surface a clear `LaunchError` with reason `launcher-missing` when neither resolves ŌĆö do NOT auto-install.
2. Build env: `{ ...process.env, WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS: '--remote-debugging-port=<port> <extraBrowserArgs...>'.trim() }`. If the var already exists in `process.env` and contains `--remote-debugging-port`, refuse rather than silently overwriting (reason `port-already-configured`).
3. Spawn with `child_process.spawn(launcher, ['start', manifestPath], { cwd: project.root, env, stdio: 'pipe', windowsHide: false })`. Stream stdout/stderr through the MCP response (or to a ring buffer the tool returns).
4. Poll the CDP probe (reuse the Phase-1 connection-robustness probe ŌĆö see memory ID 227) until success or `timeoutMs`. On success, resolve with `LaunchResult`. On timeout, kill the child and reject with reason `cdp-not-ready`.
5. Track the spawned child in a module-level `Map<number, ChildProcess>` so `stop()` and process-exit cleanup can find it.
6. Register `process.on('exit')` and `SIGINT`/`SIGTERM` handlers once (idempotent) to invoke `stop()` for any tracked children.

**Files to create:**

- `tests/launch/launchExcel.test.ts` ŌĆö mock `child_process.spawn` and the CDP probe; verify env construction, port collision rejection, timeout path, stop wiring.

## Phase 3 ŌĆö MCP tool surface

**Goal:** Expose launch/stop as MCP tools so the agent can call them.

**Files to create:**

- `src/tools/lifecycle.ts` with three `defineTool` exports:
  - `excel_detect_addin` ŌĆö params: `{ cwd?: string }`. Returns the `AddinProject` JSON or a clear "not an Excel add-in repo" message. Category: `LIFECYCLE`.
  - `excel_launch_addin` ŌĆö params:
    ```ts
    {
      cwd?: string;                   // default: server cwd
      port?: number;                  // default: from CLI --launch-port or 9222
      manifestPath?: string;          // override detection
      extraBrowserArgs?: string[];
      timeoutMs?: number;
      autoConnect?: boolean;          // default true ŌĆö call ensureBrowserConnected after launch
    }
    ```
    Behavior: detect ŌåÆ launch ŌåÆ optionally connect. Emit progress events for each phase. Idempotent: if a launch is already tracked for this manifest, return its existing `LaunchResult` instead of spawning again.
  - `excel_stop_addin` ŌĆö params: `{ manifestPath?: string }`. Looks up the tracked child and runs `office-addin-debugging stop <manifest>`, then `kill` if the stop command does not exit cleanly within 10s.

**Files to modify:**

- [src/tools/tools.ts](../../src/tools/tools.ts) ŌĆö add `import * as lifecycleTools from './lifecycle.js'` and spread into `rawTools`.
- [src/tools/categories.ts](../../src/tools/categories.ts) ŌĆö add `LIFECYCLE` enum value. Update any category-aware docs/CLI generators (run `npm run gen` and inspect the diff).
- [src/bin/excel-webview2-mcp-cli-options.ts](../../src/bin/excel-webview2-mcp-cli-options.ts) ŌĆö add CLI flags:
  - `--launch-port <number>` (default 9222) ŌĆö default port for `excel_launch_addin`.
  - `--launch-timeout <ms>` (default 60000).
  - `--auto-launch` ŌĆö if set and the working dir is detected as an Excel add-in, run `excel_launch_addin` once at server startup.

## Phase 5 ŌĆö Manual validation checklist

Not automatable in CI; run locally against a real Excel add-in repo.

1. `cd` into an add-in repo with `office-addin-debugging` installed and no Excel running.
2. Start the MCP server with `--auto-launch`. Confirm Excel opens, the taskpane loads, and `list_pages` returns the taskpane URL.
3. Restart Excel manually; confirm `excel_stop_addin` cleanly stops and that re-running `excel_launch_addin` succeeds (idempotency / cleanup).
4. Run with `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS` already set in the shell ŌĆö confirm the tool refuses with `port-already-configured`.
5. Run in a non-add-in repo ŌĆö confirm `excel_detect_addin` returns the "not an Excel add-in repo" result and `excel_launch_addin` refuses without crashing.
6. Kill the MCP server with Ctrl+C ŌĆö confirm the spawned Excel child is stopped (no orphan `office-addin-debugging` process).

## Out of scope

- Installing `office-addin-debugging` automatically. The tool surfaces a clear error and lets the user run `npm i -D office-addin-debugging` themselves.
- Non-Windows platforms. WebView2 is Windows-only; on macOS/Linux the launch tools should detect platform and refuse with a clear message.
- Managing the dev-server lifecycle independently. `office-addin-debugging start` already handles that; we do not re-implement it.
- Unified-manifest sideloading quirks beyond detection. If the unified manifest path needs different launcher args, address in a follow-up.
